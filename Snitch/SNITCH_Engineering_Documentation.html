01
Executive summary
What SNITCH is and what the project demonstrates

SNITCH is a full-stack fashion e-commerce platform built around two main roles: buyers and sellers. Buyers can register, log in, browse products, inspect product variants, manage a persistent cart, complete checkout through Razorpay test mode, and reach an order-success state after payment. Sellers can create products and manage variant-level inventory.

Core value proposition

A production-inspired MERN commerce workflow that connects authentication, catalog browsing, variant-aware inventory, server-side cart management, Redux state, and payment processing in one end-to-end application.

Engineering focus

  • Frontend architecture — feature-based React organization with Redux Toolkit and reusable hooks.
  • Backend architecture — routes, validators, middleware, controllers, DAO/service logic, and Mongoose models.
  • Commerce domain logic — product variants, stock-aware cart operations, totals, checkout, and payment-state persistence.

03
Core modules
Authentication, catalog, inventory, cart, and payments
Authentication

Registration and login establish the user session and populate frontend auth state. Protected frontend routes control navigation while backend authentication middleware protects private API operations.

Products & Inventory

Products support flexible variants such as color and size. A variant can carry its own images, price, and stock quantity, allowing inventory to be tracked at the purchasable SKU-like level rather than only at the parent product level.

Cart

The cart is persisted on the server. Buyers can add a selected product variant, increment quantity, decrement quantity, and remove an item. Cart retrieval uses MongoDB aggregation to enrich cart references with product information before returning data to the client.

Payment

The checkout flow creates payment data through the backend and opens Razorpay Checkout in test mode. Payment records move through lifecycle states such as pending and paid, and the UI presents an order-success experience after successful completion.

04
Application workflows
End-to-end buyer and seller journeys

Buyer journey

Register / Login → Product Catalog → Product Detail → Select Variant → Add to Cart → Quantity Management → Checkout → Razorpay → Payment Success → Order Success

Seller journey

Seller Login → Dashboard → Create Product → Add Variant Attributes → Upload / assign images → Set price → Set inventory quantity → Product available to buyers
Important domain decision

The cart should identify the exact selected variant, not only the parent product. This preserves correct price, image, size/color selection, and stock validation throughout checkout.

Cart aggregation

The cart read path uses MongoDB aggregation stages such as $unwind and $lookup to resolve product references and return a client-friendly cart representation. This avoids forcing the frontend to make separate product requests for each cart item.

State management strategy

Redux Toolkit stores cross-page application state such as authenticated user data and cart items. Feature service modules isolate HTTP requests, while custom hooks expose reusable UI-facing operations and loading/error behavior.

Separation of concerns

Pages focus on rendering and user interaction; hooks coordinate feature behavior; service modules call the backend; Redux slices hold shared client state. This keeps API code out of presentation components.

Design responsibilities

  • The frontend starts checkout and presents the gateway UI.
  • The backend owns secret credentials and gateway order creation.
  • Payment state is persisted so success is not represented only by temporary client state.
  • Test mode is used for project demonstration; production activation requires live credentials and production-grade verification.
⚠️

Security rule: Razorpay key secret, database URI, and JWT secret must never be exposed in frontend code or committed to Git.

Testing strategy
Recommended verification plan for the completed project

Priority test coverage

  • Authentication: valid/invalid registration and login, protected-route access.
  • Products: product creation, variant creation, invalid price/stock input.
  • Cart: duplicate add behavior, increment, decrement to boundary, remove, stock limits, total calculation.
  • Payments: order creation failure, successful test payment, duplicate callback/idempotency behavior.
Recommended pyramid

Unit tests for domain calculations and validators; integration tests for Express + MongoDB flows; a small number of end-to-end tests for login → product → cart → payment-success journey.

Engineering decisions & learning
The strongest technical lessons demonstrated by the project
  • Modeling product variants is more realistic than storing one global product price and quantity.
  • Cart state needs synchronization between backend persistence and Redux client state.
  • MongoDB aggregation can transform reference-heavy persistence models into frontend-friendly responses.
  • Payment integration is a lifecycle, not only a button: creation, pending state, gateway interaction, success handling, and persisted status all matter.
  • Feature-based frontend structure and layered backend structure make a medium-sized project easier to reason about and extend.
Placement value

SNITCH demonstrates end-to-end ownership: UI implementation, REST API design, authentication, database modeling, aggregation pipelines, state management, cart domain logic, inventory workflows, external payment integration, and debugging across the complete stack.
